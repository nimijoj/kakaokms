③ 16개 키 오퍼레이션 REST API 상세 설계
1
POST /api/v1/keys/symmetric
대칭키 생성
KMIP: Create
RFP §2-나-4
📖 SG-KMS [키 관리]-[대칭키] 생성과 동일. ARIA·AES·SEED·TDES 등 지원. 생성 즉시 state=Pre-Active. Activate API 호출 후 사용 가능.
Request Body
name필수키 이름 (영대문자·숫자·특수문자 -_ / 1~64자)
algorithm필수AES / ARIA / SEED / TDES / BLOWFISH
size필수키 길이 (비트): "128" / "192" / "256"
alias선택AlterName (미입력 시 name과 동일)
group선택소속 Agent명 (미입력 시 토큰의 agentName)
요청/응답 예문
# Request POST /api/v1/keys/symmetric Authorization: Bearer {JWT} { "name": "PAYMENT_AES256_01", "algorithm": "AES", "size": "256", "alias": "pay_main_key", "group": "PAYMENT_AGENT" } # Response { "ResultStatus": 1, "Message": "success", "Data": { "uuid": "abc18772-...", "name": "PAYMENT_AES256_01", "state": "Pre-Active", # 생성 직후 "algorithm": "AES", "size": "256", "group": "PAYMENT_AGENT", "origin_uuid": "abc18772-..." } }
2
POST /api/v1/keys/asymmetric
비대칭키 쌍 생성
KMIP: Create Key Pair
📖 SG-KMS [키 관리]-[비대칭키] 생성과 동일. RSA·ECDSA 쌍 생성. PublicKey·PrivateKey 각각 uuid 발급. parent_uuid로 쌍 연결.
Request Body
name필수키 쌍 이름 (PublicKey/PrivateKey 공통 prefix)
algorithm필수RSA / ECDSA
size필수RSA: "1024"/"2048"/"4096" / ECDSA: curve명
group선택소속 Agent명
응답 예문
# Response — 키 쌍 2개 uuid 반환 { "ResultStatus": 1, "Message": "success", "Data": { "public_key": { "uuid": "pub-uuid-...", "type": "PublicKey", "state": "Pre-Active" }, "private_key": { "uuid": "priv-uuid-...", "type": "PrivateKey", "state": "Pre-Active", "parent_uuid": "pub-uuid-..." } } }
3
POST /api/v1/keys/register
외부 키 반입 등록
KMIP: Register
반입 대체
📖 SG-KMS [정책 가져오기]와 동일. Hex 포맷 외부 키를 KMS에 등록. RFP §2-나-5 Hex 선호 명시. 기존 KMS에서 마이그레이션 시 사용.
Request Body
name필수등록할 키 이름
algorithm필수AES / ARIA / RSA 등
size필수키 길이 (비트)
value필수키 값 (Hex 문자열)
type선택SymmetricKey / PublicKey / PrivateKey
group선택소속 Agent명
요청/응답 예문
POST /api/v1/keys/register { "name": "LEGACY_AES256_IMPORT", "algorithm": "AES", "size": "256", "value": "0123456789ABCDEF0123456789ABCDEF...", "type": "SymmetricKey" } # Response { "ResultStatus": 1, "Message": "success", "Data": { "uuid": "new-uuid-...", "state": "Pre-Active" } }
4
POST /api/v1/keys/{uuid}/rekey
키 갱신
KMIP: Re-key
📖 기존 키를 새 키로 갱신. origin_uuid로 이전 키 계보 추적. sequence 증가. 이전 키는 Deactivated 전환. 새 키는 Pre-Active로 생성.
Path Variable + Request Body
uuid (Path)필수갱신할 기존 키의 uuid
new_name선택새 키 이름 (미입력 시 기존 name 유지)
응답 예문
POST /api/v1/keys/abc18772-.../rekey { "new_name": "PAYMENT_AES256_02" } # Response — 새 키 uuid 반환 { "ResultStatus": 1, "Message": "success", "Data": { "uuid": "new-uuid-xyz99900", "state": "Pre-Active", "origin_uuid": "abc18772-...", # 원본 "sequence": 1 # 버전 증가 } }
5
POST /api/v1/keys/locate
키 목록 조회
KMIP: Locate
AlterName 지원
📖 카카오뱅크 현행 API 기준 POST 방식. name 또는 alias(AlterName)로 조회. RFP §2-나-5 "클라이언트 AlterName으로 위치(Locate) 기능 제공" 충족. 클라이언트는 자신에게 매핑된 Agent의 키만 조회 가능.
Request Body (둘 중 하나 이상)
name선택키 이름으로 조회
alias선택AlterName으로 조회 — RFP 명시 요건
state선택상태 필터: Active / Deactivated 등
group선택Agent 필터
💡 모든 파라미터 생략 시 접근 가능한 전체 키 목록 반환
요청/응답 예문 (현행 샘플 기준)
# AlterName으로 조회 (RFP 명시 요건) POST /api/v1/keys/locate { "alias": "KakaoBank-Test00001" } # Response (현행 Locate 샘플 구조 동일) { "ResultStatus": 1, "Message": "success", "Data": [{ "uuid": "abc18772-14bb-4bd0-a10b-63ed81d499d4", "name": "KakaoBank-Test00001", "alias": "KakaoBank-Test00001", "algorithm": "AES", "type": "SymmetricKey", "size": "256", "group": "PAYMENT_AGENT", "state": "Active", "start_date": "2026-01-01T00:00:00", "end_date": "2027-01-01T00:00:00", "origin_uuid": "abc18772-..." }] }
6
POST /api/v1/keys/get
키 값 조회 (Hex)
KMIP: Get
ExportKey 대체
📖 카카오뱅크 현행 API 기준 POST 방식. uuid로 실제 키 값(Hex) 반환. SG-KMS 현행 GET /api/exportKey/by/name/{policyName}을 uuid 기반 POST로 대체. Deactivated 키도 복호화 목적으로 조회 가능.
Request Body
uuid필수키 uuid (Locate로 먼저 조회)
⚠️ value 필드는 이 API의 응답에서만 포함됩니다. Locate 등 다른 API는 value를 반환하지 않습니다.
응답 예문 (현행 Get 샘플 구조 동일)
POST /api/v1/keys/get { "uuid": "4f4b8491-xxxx-xxxx-xxxx-xxxxxxxxxxxx" } # Response (현행 Get 샘플 기준) { "ResultStatus": 1, "Message": "success", "Data": { "uuid": "4f4b8491-...", "name": "PUB-OP_Tester", "algorithm": "RSA", "type": "PublicKey", "size": "2048", "group": "PAYMENT_AGENT", "state": "Deactivated", "value": "3082010a02820101...", # Hex 키 값 "origin_uuid": "4f4b8491-...", "parent_uuid": null, "sequence": 0 } }
7
POST /api/v1/keys/locate
키 속성 조회
KMIP: Get Attributes
💡 SG-KMS 현행 구조에서 Get Attributes는 Locate 응답의 메타 정보로 충족됩니다. Locate 응답에 RFP §2-나-5 "모든 응답에 암호키 정보·생성/만료일·상태 등 메타 정보 포함" 요건이 이미 반영되어 있습니다.
# uuid로 특정 키의 속성만 조회할 때도 Locate POST 사용 POST /api/v1/keys/locate { "uuid": "abc18772-..." } # Response: Locate와 동일 구조 (value 제외) # → uuid/name/alias/algorithm/type/size/group/state/create_date/start_date/end_date/origin_uuid 포함
8~10
키 속성 추가 / 수정 / 삭제
KMIP: Add / Modify / Delete Attribute
📖 RFP §2-나-5 "모든 암호키 메타(Attribute) 관리(CRUD) 기능 제공 (암호키 상태/만료일 포함)". 사용자 정의 속성(Tag, 설명 등) 및 만료일·상태 변경에 활용.
POST 속성 추가
POST /api/v1/keys/{uuid}/attributes { "key": "Tag", "value": "payment-critical" } # Response { "ResultStatus": 1, "Message": "success" }
PUT 속성 수정
PUT /api/v1/keys/{uuid}/attributes { "key": "end_date", "value": "2028-01-01T00:00:00" } # Response { "ResultStatus": 1, "Message": "success" }
DELETE 속성 삭제
DELETE /api/v1/keys/{uuid} /attributes/Tag # Response { "ResultStatus": 1, "Message": "success" }
11~12
키 활성화 / 키 비활성화
KMIP: Activate / Revoke
POST /api/v1/keys/{uuid}/activate

POST /api/v1/keys/hash
해시 연산
KMIP: Hash
💡 SG-KMS Crypto API(api/crypto/enc/symmkey §3.2.5)와 동일 계층. SHA-256/384/512 등 해시 연산. 키 없이 호출 가능 (algorithm 지정).

algorithm필수SHA256 / SHA384 / SHA512
응답 예문

14
POST /api/v1/keys/using-rekey
자동 Re-key 후 키 반환
KMIP: Using Re-key
📖 SG-KMS §3.4.2 샘플코드의 agent.getKey(token, policyName)와 동일한 역할. name으로 키 조회 → 만료 확인 → 만료 시 자동 Re-key → 유효한 키 값(Hex) 반환. 키 만료 처리를 애플리케이션에서 투명하게 처리.
Request Body
name필수키 이름 (정책명과 동일 역할)
alias선택AlterName으로도 조회 가능
응답 예문
POST /api/v1/keys/using-rekey { "name": "PAYMENT_AES256_01" } # 만료 미달: 현재 키 그대로 반환 { "ResultStatus": 1, "Message": "success", "Data": { "uuid": "abc18772-...", "value": "0123456789ABCDEF...", "is_rekeyed": false } } # 만료 초과: Re-key 후 새 키 반환 { "Data": { "uuid": "new-uuid-...", "value": "FEDCBA9876543210...", "is_rekeyed": true, "origin_uuid": "abc18772-..." } }
15
GET /api/v1/mgmt/keys
전체 키 목록 조회
ADMIN 전용
RFP §2-나-5 복호화 솔루션
🔒 PORTAL_ADMIN 전용 — JWT의 role=ADMIN 확인 후에만 접근 허용. RFP §2-나-5 "등록된 모든 키 값의 조회 기능을 제공하여야 함 (별도 복호화 솔루션 목적)" 충족. value는 포함되지 않으며 각 키별로 Get API 호출 필요.
Query Parameters (선택)
state선택상태 필터: Active / Deactivated 등
group선택Agent 필터
algorithm선택알고리즘 필터: AES / RSA 등
page선택페이지 번호 (기본 1)
size선택페이지 크기 (기본 100)
응답 예문
GET /api/v1/mgmt/keys?state=Active&group=PAYMENT_AGENT Authorization: Bearer {ADMIN_JWT} { "ResultStatus": 1, "Message": "success", "Data": { "total": 1523, "page": 1, "size": 100, "keys": [ { "uuid": "abc18772-...", "name": "PAYMENT_AES256_01", "alias": "pay_main", "algorithm": "AES", "size": "256", "group": "PAYMENT_AGENT", "state": "Active", "start_date": "2026-01-01", "end_date": "2027-01-01" # value 없음 — Get API로 별도 조회 }, # ... 더 많은 키 ] } }
16
DELETE /api/v1/mgmt/keys/{uuid}
암호키 삭제
ADMIN 전용
🔒 PORTAL_ADMIN 전용. Deactivated 상태에서만 삭제 가능. Active 키는 먼저 Revoke 후 삭제. 삭제 후 복구 불가 — 운영 정책상 이중 확인 절차 권장.
Path Variable
uuid (Path)필수삭제할 키의 uuid. Deactivated 상태여야 함.
⚠️ Active 상태 키 삭제 시도 → ResultStatus: 0, errorCode: 409 반환
응답 예문
# Deactivated 키 삭제 (성공) DELETE /api/v1/mgmt/keys/abc18772-... Authorization: Bearer {ADMIN_JWT} { "ResultStatus": 1, "Message": "success", "Data": { "uuid": "abc18772-...", "state": "Destroyed" } } # Active 키 삭제 시도 (실패) { "ResultStatus": 0, "Message": "Key must be Deactivated before deletion", "errorCode": 409 }
④ 공통 오류 응답 — ResultStatus: 0
📋 SG-KMS 현행 API 오류 응답 구조(errorCode/errorId/message §3.2.3)를 유지하되 ResultStatus를 추가로 반환하여 카카오뱅크 현행 구조와 일치시킵니다.
# 인증 실패 (401) { "ResultStatus": 0, "Message": "Unauthorized", "errorCode": 401, "errorId": "AUTH_FAILED", "Data": null } # 접근 권한 없음 (403) { "ResultStatus": 0, "Message": "Key not accessible for this client", "errorCode": 403, "errorId": "ACCESS_DENIED", "Data": null }
# 키 없음 (404) { "ResultStatus": 0, "Message": "Key not found", "errorCode": 404, "errorId": "KEY_NOT_FOUND", "Data": null } # 상태 충돌 (409) — Active 키 삭제 시도 등 { "ResultStatus": 0, "Message": "Invalid state transition", "errorCode": 409, "errorId": "INVALID_STATE", "Data": null }

② SG-KMS 내부 객체 → 카카오뱅크 REST API 레이어 매핑 설계
💡 설계 원칙: SG-KMS의 내부 구조를 변경하지 않고 그대로 유지합니다. 카카오뱅크 REST API 레이어는 SG-KMS 위에 올라가는 추상화 계층으로, SG-KMS 객체를 REST 자원으로 재해석합니다.
SG-KMS 내부 객체 SG-KMS 관리 메뉴 REST API 자원명 매핑 설명
KMS API 계정
id + password		 [KMS API 관리]-[KMS API 계정] Client (클라이언트) KMS API 계정의 아이디가 클라이언트 식별 이름이 됩니다. 비밀번호는 내부 인증에만 사용되며 REST API에는 노출되지 않습니다.
Agent
agentName		 [보안 정책]-[Agent 관리] Client의 서비스 식별자 agentName이 클라이언트 이름 인증의 핵심 식별자입니다. REST API 토큰에 클라이언트명이 포함되어 KMS API 계정 + Agent를 하나의 Client 개념으로 통합합니다.
정책(Policy)
policyName → keyValue		 [보안 정책]-[정책 관리] KeyGroup (키 그룹) 정책이 키에 대한 접근 경로를 정의하듯, 키 그룹이 클라이언트의 키 접근 단위가 됩니다. 정책명 기반 접근 → 그룹명 기반 접근으로 전환합니다.
대칭키 그룹(SymKeyGroup)
여러 대칭키 묶음		 [키 관리]-[대칭키 그룹] KeyGroup 내 복수 키 SG-KMS의 대칭키 그룹은 BA-SCP 전용이므로 사용하지 않습니다. Agent를 키 그룹으로 활용하며, Agent에 적용된 정책(Policy) 단위로 키에 접근합니다.
정책-Agent 적용
N:M 매핑		 [보안 정책]-[정책-Agent 적용] Client-Group 권한 매핑 SG-KMS 정책-Agent 적용 구조를 그대로 활용합니다. 클라이언트(KMS API 계정)는 agentName으로 지정된 Agent에 적용된 정책(키)에만 접근 가능하며, 1:N으로 복수 Agent 접근 권한 부여가 가능합니다.
암호화 키(Key)
AES/ARIA/RSA 등		 [키 관리]-[대칭키]/[비대칭키] CryptoKey (암호키) SG-KMS 키가 KMIP 기반 uuid, state, alias, zone 등의 속성을 가진 CryptoKey 자원으로 표현됩니다. 현행 API 응답 필드명(uuid, state, alias 등) 그대로 사용합니다.
③ 클라이언트 이름 인증 설계 — agentName 기반 Bearer Token
📖 매뉴얼 근거 §3.2.3: SG-KMS 현행 인증은 Basic Auth(KMS API 계정 ID:PW)와 agentName 파라미터를 결합하여 JWT를 발급합니다. JWT에는 name(계정명), agentid, agentname, role 등이 포함됩니다. 카카오뱅크 설계는 이 agentName 중심 인증 철학을 그대로 계승합니다.
🔵 현행 SG-KMS 인증 방식 (§3.2.3.1)
# KMS API 계정 + agentName으로 토큰 발급 POST /api/auth/login/basic Authorization: Basic {base64(계정ID:비밀번호)} { "agentName": "PAYMENT_AGENT" } # 발급된 JWT 페이로드 내용 { "name": "kms_api_account", # KMS API 계정명 "agentid": "a6af0f0b-...", # Agent UUID "agentname": "PAYMENT_AGENT", # Agent 명 (식별자) "role": "api", # 역할 "iss": "penta", "exp": 1701671418 # 만료 시각 }
🟢 카카오뱅크 REST API 설계 인증 방식
# 클라이언트 이름을 식별자로 한 토큰 발급 POST /api/v1/auth/token { "clientName": "PAYMENT_SVC", # agentName과 동일 역할 "credential": "..." # 내부 인증 정보 } # 발급되는 JWT 페이로드 설계 { "clientName": "PAYMENT_SVC", # 클라이언트 식별명 "clientId": "uuid-...", # 내부 클라이언트 ID "role": "CLIENT", # CLIENT | ADMIN "iss": "kakaobank-kms", "exp": 1700000000 }
✅ 설계 연속성: SG-KMS의 agentName이 카카오뱅크 설계에서는 clientName이 됩니다. SG-KMS 내부적으로는 KMS API 계정 + agentName 조합이 그대로 사용되며, REST API 레이어에서만 클라이언트 이름 단일 식별자로 추상화됩니다.
카카오뱅크 클라이언트명 규칙: 서비스별 고유명 (예: PAYMENT_SVC, ACCOUNT_SVC, CARD_SVC)
📋 클라이언트 등록 → 토큰 발급 전체 흐름
1
관리자: 클라이언트 등록 — POST /api/v1/mgmt/clients
PORTAL_ADMIN이 KMS API 계정(클라이언트명, IP ACL, role)을 등록합니다. SG-KMS 현행 구조 그대로 KMS API 계정과 Agent는 각각 독립적으로 생성합니다. 클라이언트 등록 후 별도로 Agent(그룹)를 생성하고 정책을 적용한 뒤 권한을 부여합니다.
2
서비스: 토큰 발급 — POST /api/v1/auth/token
클라이언트명(PAYMENT_SVC)과 접근할 agentName으로 JWT 발급 요청. SG-KMS 내부에서는 KMS API 계정(ID/PW) + agentName으로 Basic Auth 처리. 해당 Agent에 적용된 정책(키)에만 접근 가능한 토큰이 발급됩니다.
3
서비스: API 호출 — Authorization: Bearer {JWT}
발급된 JWT로 키 오퍼레이션 API(Locate, Get, Create 등)를 호출합니다. JWT의 clientName으로 접근 가능한 그룹과 키를 필터링합니다.

발급된 JWT 페이로드 구조
# JWT를 Base64 디코딩하면 아래 페이로드 확인 가능 { "name": "kms_api_user", # KMS API 계정 ID "agentid": "a6af0f0b-14bb-4bd0-a10b-63ed81d499d4", # Agent UUID "agentname": "PAYMENT_AGENT", # 요청한 agentName — 접근 가능 정책 범위 결정 "role": "api", # 역할 "iss": "penta", "iat": 1701667818, # 발급 시각 (Unix timestamp) "exp": 1701671418 # 만료 시각 (1시간 유효) }
💡 핵심: JWT의 agentname 필드가 접근 가능한 정책(키)의 범위를 결정합니다. PAYMENT_AGENT에 적용된 정책만 ExportKey·Crypto API로 호출할 수 있습니다.

정책명으로 키 값(Hex) 조회
📖 Auth API에서 발급받은 JWT 토큰을 Bearer 헤더에 담고, URL 경로에 정책명을 지정하면 해당 정책의 키 값을 Hex 문자열로 반환합니다. agentName에 해당 정책이 적용되어 있어야만 접근 가능합니다.

✅ 카카오뱅크 현행 사용 방식: 이 API가 현재 카카오뱅크 시스템에서 암호화 키를 꺼내는 핵심 호출입니다. 애플리케이션이 이 Hex 키 값을 받아 직접 암복호화에 사용합니다.
⑤ Crypto API — 암호화·복호화 직접 처리 (매뉴얼 §3.2.5)
💡 ExportKey가 "키 값을 꺼내서 애플리케이션이 직접 암복호화"하는 방식이라면, Crypto API는 "데이터를 KMS에 보내서 KMS가 직접 암복호화를 수행"하는 방식입니다. 두 방식 모두 동일한 인증 토큰을 사용합니다.

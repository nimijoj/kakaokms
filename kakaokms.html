질의 예상 1순위
IP ACL 예외 설정 미지원
RFP §2-나-5 명시 요건 · 대응 난이도 ★★★
📌 RFP 원문
"IP ACL 예외 설정 가능" — 특정 IP를 허용 범위에서 제외하는 예외 차단 기능 명시 요구
✅ 현재 제공 범위
단일 IP, CIDR(10.10.1.0/24), 범위(10.0.0.1~254) ACL CRUD 완전 지원. 예외 설정은 미지원.
💬 선제 대응 스크립트
"단일·CIDR·범위 IP ACL은 완전히 지원됩니다. 예외 설정 기능은 현재 미지원 상태입니다. 실제 운영에서 예외 IP 설정이 필요했던 구체적인 케이스가 있으신지 먼저 여쭤봐도 될까요? 케이스를 확인한 후 다음 버전 로드맵 반영 여부를 제품팀과 협의해 드리겠습니다."
⚡ 역질문 포인트
예외 설정이 실제로 활용된 사례가 있는지 확인. 없다면 "운영 실적이 없는 기능"으로 협의 범위 축소 가능.
질의 예상 2순위
PEM·PKCS 반출입 포맷
RFP §2-나-7 명시 요건 · 대응 난이도 ★★
📌 RFP 원문
"Hexadecimal, PKCS#1, PKCS#8, KEK, Component, Base64 등" 다양한 포맷 지원 요구
✅ 현재 제공 범위
Hex(Get API 대체) · Component(ec1/ec2 완전 지원). PEM·PKCS·KEK·Base64는 별도 Export API 없음.
💬 선제 대응 스크립트
"RFP §2-나-5에서 Hexa Decimal을 선호로 직접 명시하셨고, 현행 카카오뱅크 API도 Hex 기반으로 확인되어 Hex로 설계했습니다. Component 포맷도 분할 반출입 API로 완전 지원됩니다. 실제 업무에서 PEM·PKCS 파일 형태로 반출이 필요한 시스템이 있으신지 확인하겠습니다. 있다면 별도 Export API 추가 개발로 협의 가능합니다."
⚡ 역질문 포인트
현행 시스템에서 PEM·PKCS 포맷을 실제로 사용하는 시스템이 있는지 확인. 없다면 Hex로 충분히 대체 가능 논거 강화.

✅ 현재 제공 범위
웹 UI 이력 조회 화면 제공 → RFP 명시 요건(웹 UI) 완전 충족. 이력 API는 협의 항목으로 미포함.
💬 선제 대응 스크립트
"반출입 이력은 웹 UI 화면을 통해 완전히 제공됩니다. RFP §2-나-8에 명시된 웹 UI 요건은 충족됩니다. 별도 신청 시스템과 자동화 연동이 필요하신 경우, 이력 조회 API는 협의 항목으로 추가 개발이 가능합니다. 연동 필요 여부와 범위를 확인해 주시면 별도 제안드리겠습니다."
⚡ 역질문 포인트
RFP에 (협의)로 명시된 항목임을 먼저 언급. 자동화 연동이 필요한 신청 시스템의 존재 여부 확인.

② SG-KMS 내부 객체 → 카카오뱅크 REST API 레이어 매핑 설계
💡 설계 원칙: SG-KMS의 내부 구조를 변경하지 않고 그대로 유지합니다. 카카오뱅크 REST API 레이어는 SG-KMS 위에 올라가는 추상화 계층으로, SG-KMS 객체를 REST 자원으로 재해석합니다.
SG-KMS 내부 객체 SG-KMS 관리 메뉴 REST API 자원명 매핑 설명
KMS API 계정
id + password		 [KMS API 관리]-[KMS API 계정] Client (클라이언트) KMS API 계정의 아이디가 클라이언트 식별 이름이 됩니다. 비밀번호는 내부 인증에만 사용되며 REST API에는 노출되지 않습니다.
Agent
agentName		 [보안 정책]-[Agent 관리] Client의 서비스 식별자 agentName이 클라이언트 이름 인증의 핵심 식별자입니다. REST API 토큰에 클라이언트명이 포함되어 KMS API 계정 + Agent를 하나의 Client 개념으로 통합합니다.
정책(Policy)
policyName → keyValue		 [보안 정책]-[정책 관리] KeyGroup (키 그룹) 정책이 키에 대한 접근 경로를 정의하듯, 키 그룹이 클라이언트의 키 접근 단위가 됩니다. 정책명 기반 접근 → 그룹명 기반 접근으로 전환합니다.
대칭키 그룹(SymKeyGroup)
여러 대칭키 묶음		 [키 관리]-[대칭키 그룹] KeyGroup 내 복수 키 SG-KMS의 대칭키 그룹(복수 키 묶음)을 REST API의 키 그룹 내 키 배열로 표현합니다. groupType:SHARED가 대칭키 그룹 성격에 해당합니다.
정책-Agent 적용
N:M 매핑		 [보안 정책]-[정책-Agent 적용] Client-Group 권한 매핑 정책에 Agent를 적용하는 N:M 구조 → 클라이언트에게 그룹 접근 권한 부여(1:N)로 단순화합니다. DEFAULT 그룹은 클라이언트의 전용 정책과 동일한 역할입니다.
암호화 키(Key)
AES/ARIA/RSA 등		 [키 관리]-[대칭키]/[비대칭키] CryptoKey (암호키) SG-KMS 키가 KMIP 기반 uuid, state, alias, zone 등의 속성을 가진 CryptoKey 자원으로 표현됩니다. 현행 API 응답 필드명(uuid, state, alias 등) 그대로 사용합니다.
③ 클라이언트 이름 인증 설계 — agentName 기반 Bearer Token
📖 매뉴얼 근거 §3.2.3: SG-KMS 현행 인증은 Basic Auth(KMS API 계정 ID:PW)와 agentName 파라미터를 결합하여 JWT를 발급합니다. JWT에는 name(계정명), agentid, agentname, role 등이 포함됩니다. 카카오뱅크 설계는 이 agentName 중심 인증 철학을 그대로 계승합니다.
🔵 현행 SG-KMS 인증 방식 (§3.2.3.1)
# KMS API 계정 + agentName으로 토큰 발급 POST /api/auth/login/basic Authorization: Basic {base64(계정ID:비밀번호)} { "agentName": "PAYMENT_AGENT" } # 발급된 JWT 페이로드 내용 { "name": "kms_api_account", # KMS API 계정명 "agentid": "a6af0f0b-...", # Agent UUID "agentname": "PAYMENT_AGENT", # Agent 명 (식별자) "role": "api", # 역할 "iss": "penta", "exp": 1701671418 # 만료 시각 }
🟢 카카오뱅크 REST API 설계 인증 방식
# 클라이언트 이름을 식별자로 한 토큰 발급 POST /api/v1/auth/token { "clientName": "PAYMENT_SVC", # agentName과 동일 역할 "credential": "..." # 내부 인증 정보 } # 발급되는 JWT 페이로드 설계 { "clientName": "PAYMENT_SVC", # 클라이언트 식별명 "clientId": "uuid-...", # 내부 클라이언트 ID "role": "CLIENT", # CLIENT | ADMIN "defaultGroup": "PAYMENT_SVC_GRP", "iss": "kakaobank-kms", "exp": 1700000000 }
✅ 설계 연속성: SG-KMS의 agentName이 카카오뱅크 설계에서는 clientName이 됩니다. SG-KMS 내부적으로는 KMS API 계정 + agentName 조합이 그대로 사용되며, REST API 레이어에서만 클라이언트 이름 단일 식별자로 추상화됩니다.
카카오뱅크 클라이언트명 규칙: 서비스별 고유명 (예: PAYMENT_SVC, ACCOUNT_SVC, CARD_SVC)
📋 클라이언트 등록 → 토큰 발급 전체 흐름
1
관리자: 클라이언트 등록 — POST /api/v1/mgmt/clients
PORTAL_ADMIN이 클라이언트명, IP ACL, role을 등록합니다. SG-KMS에서는 KMS API 계정 + Agent를 각각 생성하는 과정에 해당합니다. 등록 시 DEFAULT 그룹(PAYMENT_SVC_GRP)이 자동 생성됩니다.
2
서비스: 토큰 발급 — POST /api/v1/auth/token
클라이언트명(PAYMENT_SVC)으로 JWT 발급 요청. SG-KMS 내부에서는 agentName=PAYMENT_SVC로 Basic Auth 처리. 발급된 JWT에 clientName, role, defaultGroup이 포함됩니다.
3
서비스: API 호출 — Authorization: Bearer {JWT}
발급된 JWT로 키 오퍼레이션 API(Locate, Get, Create 등)를 호출합니다. JWT의 clientName으로 접근 가능한 그룹과 키를 필터링합니다.

⑤ 키를 그룹(Agent)으로 매핑하는 설계 — 정책-Agent 적용 구조 활용
📖 매뉴얼 근거 §1.5.2, §1.5.3: SG-KMS에서 키를 그룹(Agent)에 매핑한다는 것은 "키를 참조하는 정책을 생성하고 그 정책을 Agent에 적용"하는 것입니다. 하나의 Agent에 복수의 정책을 적용함으로써 Agent 단위로 여러 키에 대한 접근을 관리합니다. REST API 설계에서는 이것을 "그룹에 키 추가/제거" API로 추상화합니다.
SG-KMS: 정책 생성 → Agent 적용 (§1.5.2, §1.5.3)
Step 1. [키 관리]에서 대칭키(AES256_PAY_1) 생성
Step 2. [정책 관리]에서 정책 생성 시 해당 키 선택
Step 3. [정책-Agent 적용]에서 생성된 정책을 PAYMENT_AGENT에 적용

📋 키 생성 → 그룹(Agent) 배정 → 공유 그룹 접근 권한 부여 전체 흐름
1
키 생성 — POST /api/v1/keys/symmetric
SG-KMS [키 관리]-[대칭키] 생성과 동일. 내부적으로 SG-KMS 대칭키 생성 + 해당 그룹(Agent)용 정책 자동 생성. group 파라미터로 생성 즉시 그룹 배정 가능.
POST /api/v1/keys/symmetric Authorization: Bearer {ADMIN_JWT} { "name": "AES256_PAY_3", "algorithm": "AES", "size": "256", "group": "PAYMENT_AGENT", "zone": "prod" } # 내부: SG-KMS 대칭키 생성 → PAY_KEY_POL_04 정책 자동 생성 → PAYMENT_AGENT에 적용 Response: { "ResultStatus": 1, "Data": { "uuid": "xyz99900-...", "state": "Pre-Active" } }
2
그룹에 기존 키 추가 — POST /api/v1/mgmt/groups/{groupId}/keys
SG-KMS [정책-Agent 적용]에서 Agent에 정책을 추가 적용하는 것과 동일. 배열로 복수 키 한 번에 추가 가능.
POST /api/v1/mgmt/groups/PAYMENT_AGENT/keys Authorization: Bearer {ADMIN_JWT} { "keyIds": ["abc18772-...", "bcd29883-..."] } # 내부: 각 uuid에 해당하는 정책을 PAYMENT_AGENT에 일괄 적용 Response: { "ResultStatus": 1, "Message": "success" }
3
공유 그룹 접근 권한 부여 — POST /api/v1/mgmt/clients/{id}/groups/{groupId}
SG-KMS에서 SHARED_AGENT의 접근 허용 계정에 다른 KMS API 계정을 추가하는 것과 동일. 권한 부여 후 해당 클라이언트는 SHARED_AGENT에 적용된 모든 정책(키)에 접근 가능.
POST /api/v1/mgmt/clients/ACCOUNT_SVC/groups/SHARED_AGENT Authorization: Bearer {ADMIN_JWT} # 내부: SHARED_AGENT의 접근 허용 계정에 kms_acct_id 추가 # → ACCOUNT_SVC가 agentName=SHARED_AGENT로 토큰 발급 후 공유 키 접근 가능 Response: { "ResultStatus": 1, "Message": "success" }
⑥ 실제 API 호출 시나리오 — PAYMENT_SVC의 전체 흐름
A. 운영 시나리오 — 결제 서비스 암호키 조회
# 1. 클라이언트 이름 인증 → 토큰 발급 POST /api/v1/auth/token { "clientName": "PAYMENT_SVC" } → JWT: role=CLIENT, defaultGroup=PAYMENT_SVC_GRP # 2. 키 이름으로 uuid 조회 (Locate) POST /api/v1/keys/locate Authorization: Bearer {JWT} { "name": "AES256_PAY_1" } → [{ "uuid": "abc18772-...", "state": "Active", "group": "PAYMENT_SVC_GRP" }] # 3. uuid로 키 값 조회 (Get → ExportKey 대체) POST /api/v1/keys/get Authorization: Bearer {JWT} { "uuid": "abc18772-..." } → { "uuid": "abc18772-...", "value": "0123456789ABCDEF..." } # Hex
B. 키 만료 시나리오 — Using Re-key 자동 처리
# end_date 초과 시 자동 Re-key POST /api/v1/keys/using-rekey Authorization: Bearer {JWT} { "name": "AES256_PAY_1" } # 만료 미만이면 현재 uuid 반환 → { "uuid": "abc18772-...", "isRekeyed": false } # 만료 초과이면 Re-key 후 새 uuid 반환 → { "uuid": "xyz99900-...", "isRekeyed": true, "origin_uuid": "abc18772-..." } # 계보 유지 # SG-KMS 내부에서는: # Re-key API → origin_uuid로 이전 키 추적 # sequence 증가 (버전 관리)
C. 공유 그룹 시나리오 — 여러 서비스 공통 키 접근
# 관리자: SHARED_COMMON 그룹 생성 POST /api/v1/mgmt/groups Authorization: Bearer {ADMIN_JWT} { "name": "SHARED_COMMON", "groupType": "SHARED" } # 관리자: ACCOUNT_SVC에 SHARED_COMMON 접근 권한 부여 POST /api/v1/mgmt/clients/ACCOUNT_SVC/groups/SHARED_COMMON Authorization: Bearer {ADMIN_JWT} # SG-KMS: 정책-Agent 적용 (§1.5.3) 과 동일 # ACCOUNT_SVC가 공유 키 조회 POST /api/v1/keys/locate Authorization: Bearer {ACCOUNT_JWT} { "name": "AES256_SHARED_1" } → SHARED_COMMON 그룹 키 정상 접근 ✓
D. 복호화 솔루션 시나리오 — 전체 키 일괄 조회
# PORTAL_ADMIN 토큰 발급 POST /api/v1/auth/token { "clientName": "PORTAL_ADMIN" } → JWT: role=ADMIN # 모든 키 목록 조회 (복호화 솔루션 목적) # RFP §2-나-5 명시 요건 GET /api/v1/mgmt/keys Authorization: Bearer {ADMIN_JWT} → [{uuid, name, state, group, algorithm...}, ...] # 전체 키 메타 반환 (value 제외) # value는 각 키별로 GET /keys/get으로 개별 조회
⑦ 공통 응답 구조 — 현행 SG-KMS API 응답 구조 계승
💡 매뉴얼 근거 §3.2.4: SG-KMS 현행 KMS API(ExportKey 등)의 응답은 name, value, agentName, errorCode, errorId, message 구조를 사용합니다. 카카오뱅크 현행 API 샘플(Locate·Get 응답)은 ResultStatus, Message, Data 구조를 사용하며, 이를 표준으로 채택합니다.
✅ 성공 응답 표준 (카카오뱅크 현행 API 기준)
{ "ResultStatus": 1, # 1 = 성공 "Message": "success", "Data": { # 단건: object "uuid": "abc18772-...", "name": "AES256_PAY_1", "alias": "pay_main", # AlterName "state": "Active", # status 아님! "algorithm": "AES", "size": "256", "group": "PAYMENT_SVC_GRP", "zone": "prod", "origin_uuid": "..." # Re-key 계보 } }
❌ 오류 응답 표준
{ "ResultStatus": 0, # 0 = 오류 "Message": "Unauthorized", "errorCode": 401, "errorId": "AUTH_FAILED", "Data": null } # HTTP 상태코드도 함께 반환 # 200 OK / 400 Bad Request # 401 Unauthorized / 403 Forbidden # 404 Not Found / 500 Internal Error
⚠️ 주의: 키 상태 필드는 state를 사용합니다. 현행 SG-KMS 응답 샘플에서 status가 아닌 state로 확인됨. Active·Pre-Active·Deactivated·Destroyed 4가지 상태.
